# v6.1.1 Release Notes — Buzzer Accessibility actually works

## Summary

Fixes the Accessibility grant for "QLab Buzzer.app" not being honoured at runtime in v6.1.0. The LaunchAgent now starts the .app via Launch Services so macOS associates the running process with the .app's bundle identity.

## Root cause

v6.1.0's LaunchAgent ran the bundled binary directly (`Contents/MacOS/QLab Buzzer`). macOS TCC tracks Accessibility per-app-bundle and only honours the grant when the process is launched through Launch Services. Direct binary execution bypassed that and the buzzer remained "not trusted" even with the .app toggled on.

## Fix

- **`buzzer/install_service.sh`** — LaunchAgent now uses `/usr/bin/open -W -a "/path/to/QLab Buzzer.app"`. `open` registers the launch with Launch Services; `-W` blocks until the .app exits, so launchd's `KeepAlive` still respawns on crash.
- **`buzzer/qlab_buzzer.py`** — when frozen, redirect `stdout`/`stderr` to `/tmp/qlab-buzzer.log`. `--windowed` PyInstaller bundles detach the standard streams, so `open -a` couldn't capture logs. The app now writes its own log directly.

## Upgrade

```bash
cd ~/chart-toppers && git pull && ./buzzer/install_service.sh
```

The Accessibility grant from v6.1.0 ("QLab Buzzer" in the list) is preserved — no need to re-add. After install, the trust warning should be gone from `/tmp/qlab-buzzer.log`.
