# v7.3.0 Release Notes — Buzzer Permission Hardening

**Released: May 2, 2026**

## What's New

### Buzzer keypress canary on the dashboard
- **Summary**: The footer buzzer indicator now shows live keypress count and turns amber if the daemon is alive but receiving no input.
- **Details**: The daemon counts every accepted mapped keypress and ships the running total in each heartbeat. The server flags `inputBlocked` after 30s of "connected but zero keys"; the dashboard shows an amber pulsing dot and the label "No input — check Mac permissions".
- **Impact**: Catches the silent macOS Input Monitoring revocation failure mode before a show, not during.

### Setup script wipes stale Python permissions first
- **Summary**: `setup-buzzer-host.sh` now runs `tccutil reset` for `org.python.python` before opening Privacy & Security.
- **Details**: macOS toggling a permission OFF leaves a "denied" record that silently overrides any future "allowed" record (real bug we hit on machine 216). Resetting clears both grants and denials so the user starts from a clean slate.
- **Impact**: New-machine setup is reliable on the first try.

## Technical Changes
- `buzzer/qlab_buzzer.py`: module-level `_keys_seen_total`; heartbeat body now `{"keys_seen": N}`.
- `src/server.js`: `/api/buzzer/heartbeat` parses keys_seen, tracks `buzzerConnectedAt` and `buzzerKeysSeenChangedAt`; `buzzerStatusPayload()` returns `{connected, keysSeen, inputBlocked}`; status emit on every heartbeat.
- `public/app.js`: `updateBuzzerUI` accepts the new payload shape and renders three states (disconnected / warning / connected with count).
- `public/styles.css`: amber pulsing `.buzzer-dot.warning` style.
- `scripts/setup-buzzer-host.sh`: `tccutil reset` step.
