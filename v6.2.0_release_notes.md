# v6.2.0 Release Notes — Buzzer .app distributed as a release asset

## Summary

The buzzer is no longer rebuilt on each show machine. The dev machine builds `QLab Buzzer.app` once, ships it as a release asset, and `install_service.sh` downloads the **same binary** for every install. Same binary → same code-signing fingerprint → **the macOS Accessibility grant is one-time-per-Mac and persists forever** across all subsequent chart-toppers updates (until the buzzer source itself is changed and a new release is cut).

## Why this matters

Previously every install built a fresh `.app` via PyInstaller, which produced a new ad-hoc signature each time. macOS TCC ties the Accessibility grant to the .app's code hash, so every update silently invalidated the grant and the buzzer would stop working until the user manually re-added the .app to Accessibility.

## Changes

- **`buzzer/install_service.sh`** — now uses `gh release download` to fetch `QLab Buzzer.app.zip` from the latest GitHub release and unzips it to `~/Applications/QLab Buzzer.app`. No PyInstaller, no venv, no rebuild.
- **`gh` CLI is now a hard requirement** on show machines (already used by the all-in-one updater for the private repo). The script bails with a clear message if it's missing or unauthenticated.
- **Quarantine xattr is cleared** on the unzipped .app so it launches without a Gatekeeper prompt.
- **Build flow** lives on the dev machine: `buzzer/build_app.sh` builds the .app, then it's zipped with `ditto -c -k --keepParent` and attached to the release with `gh release upload`.

## Upgrade

On every show machine:

```bash
cd ~/chart-toppers && git pull && ./buzzer/install_service.sh
```

After this install, grant Accessibility to `~/Applications/QLab Buzzer.app` once. Future buzzer-source updates will require a fresh grant, but routine chart-toppers updates that don't touch the buzzer won't invalidate the permission.
