# v6.0.5 Release Notes — Buzzer LaunchAgent self-locating plist

## Summary

Fixes the buzzer service failing to start on login when the chart-toppers repo is cloned to a non-default location (e.g. `~/TwistedMelon/chart-toppers` via the all-in-one updater instead of `~/chart-toppers`).

## Root cause

`buzzer/com.twistedmelon.qlab-buzzer.plist` hardcoded `cd ~/chart-toppers && ./buzzer/venv/bin/python3 ./buzzer/qlab_buzzer.py`. The previous `install_service.sh` just copied that static plist into `~/Library/LaunchAgents/`, so on machines where the repo lives anywhere other than `~/chart-toppers`, the LaunchAgent would `cd` into a missing directory at every boot and the buzzer would never start.

## Fix

`install_service.sh` now **generates** the plist at install time with absolute paths resolved from the actual buzzer install directory:

- `ProgramArguments` runs the venv Python and `qlab_buzzer.py` by absolute path
- `WorkingDirectory` is set to the resolved buzzer dir
- `KeepAlive` + `RunAtLoad` retained so the buzzer starts on login and restarts on crash
- Sanity checks: install bails if the venv or `qlab_buzzer.py` isn't where it expects

The static `com.twistedmelon.qlab-buzzer.plist` file in the repo is now legacy reference only — the installed plist is generated.

## Upgrade

On any machine where the buzzer is silent after reboot:

```bash
cd ~/TwistedMelon/chart-toppers && git pull && ./scripts/update-dockers.sh
```

The updater rebuilds the Docker stack and re-runs `buzzer/install_service.sh`, which now writes a correct plist. After install, confirm with:

```bash
launchctl list | grep qlab-buzzer
tail -f /tmp/qlab-buzzer.log
```

Then reboot to confirm it auto-starts.
