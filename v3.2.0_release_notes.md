# v3.2.0 Release Notes — QLab Playback Countdown

**Released: March 20, 2026**

## What's New

### Live Countdown from QLab Playback
- **Summary**: Real-time countdown display driven by QLab cue playback position
- **Details**: When a team starts playing, the bridge now polls QLab via TCP for the cue's elapsed time and feeds it back to the dashboard. The score card switches into countdown mode showing seconds remaining, with amber/warning/critical urgency states as time runs low.
- **Impact**: Operators and audiences can now see a live countdown of each team's earned time as their song plays in QLab

### Bidirectional OSC Bridge (TCP + UDP)
- **Summary**: Bridge upgraded from fire-and-forget UDP to full bidirectional communication
- **Details**: Added TCP connection to QLab with SLIP-framed OSC, inbound message parsing, /poll/start and /poll/stop HTTP endpoints, and automatic reconnection. UDP is still used for outbound commands (loadActionAt, stop, text).
- **Impact**: Enables any future feature that needs to read state back from QLab, not just send commands

### Countdown UI with Urgency States
- **Summary**: Dashboard score cards animate through countdown states
- **Details**: New CSS countdown-mode class with amber baseline, warning pulse at 15s, and critical red pulse at 5s. Progress bar drains from full to empty during playback. Score card automatically restores to earned-time display when playback stops.
- **Impact**: Clear visual feedback for operators and audience screens

## Technical Changes
- Bridge server now uses `net` (TCP) alongside `dgram` (UDP) for QLab communication
- SLIP encoding/decoding added for QLab's TCP OSC protocol
- New `/api/qlab-playback` endpoint on main server receives bridge callbacks
- Docker Compose updated with `CALLBACK_URL` and `POLL_INTERVAL_MS` environment variables
- Launch config updated with environment variable defaults
